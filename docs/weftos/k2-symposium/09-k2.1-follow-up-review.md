## 1. Executive Summary

K2.1 was the implementation bridge between the K2 Symposium's design decisions and
K3 development. Its scope was deliberately narrow: land the breaking changes and
quick wins before K3 begins, so K3 starts on a correct API foundation.

**Outcome**: K2.1 is substantially complete. The core deliverables -- SpawnBackend
enum (C1+C8), MessageTarget expansion (D19), ServiceEntry model (D1), service
routing (D19), GovernanceGate verification, and documentation updates -- are all
implemented and tested. The only K2-scoped item that did not land is post-quantum
dual signing (C6), which was blocked by the rvf-crypto crate's public API not
exposing the DualKey path. This was correctly identified as a risk in the SPARC
plan and handled as a documented gap rather than a blocker.

Final test count: 397 kernel tests (up from 373 pre-symposium), exceeding the
SPARC plan's target of 388+.

## 5. DualKey Gap (C6 -- Post-Quantum Dual Signing)

### Status: GAP (Known, Documented, Non-Blocking)

**What was planned**: Enable rvf-crypto's `DualKey` path so chain entries carry
both Ed25519 and ML-DSA-65 (Dilithium) signatures. The symposium described this
as a "quick win" (C6, ~20 lines).

**What happened**: The K2.1 SPARC plan's Task 2 pseudocode correctly included
a conditional:

```
IF DualKey::generate() exists AND dual_sign_segment() exists:
  MODIFY save_to_rvf() ...
ELSE:
  DOCUMENT gap -- what rvf-crypto actually exports for DualKey
  CREATE issue for rvf-crypto to expose the API
  SKIP implementation until API is available
```

Investigation found that rvf-crypto does not expose `DualKey::generate()` or
`dual_sign_segment()` in its public API. The crate's internal modules may
contain ML-DSA-65 code, but it is not available for consumers. The K2.1 SPARC
plan exit criteria correctly marked this as `[ ] (API unavailable)` and
noted it as a documented gap.

**What exists in the kernel today**: Zero DualKey code. Chain entries use the
standard signing path. The `rvf-crypto` crate is an optional dependency
(behind the `exochain` feature) and is used for lineage derivation, but
not for dual signing.

### Remediation Plan

1. **rvf-crypto PR**: Open a PR against rvf-crypto to expose the DualKey API:
   - `DualKey::generate() -> DualKey` (Ed25519 + ML-DSA-65 keypair)
   - `dual_sign_segment(key: &DualKey, segment: &[u8]) -> DualSignature`
   - `verify_dual(sig: &DualSignature, segment: &[u8]) -> Result<()>`
   - Accept either signature for backwards compatibility

2. **Kernel integration** (once rvf-crypto ships the API):
   - Modify `chain.rs` `save_to_rvf()` to use `dual_sign_segment()`
   - Modify `load_from_rvf()` to verify with either key
   - Add tests: `chain_entry_has_dual_signature`, `chain_verify_accepts_either_signature`, `dual_key_round_trip`
   - Estimated scope: ~25 lines of kernel code + 3 tests

3. **Target phase**: K3 (earliest) or whenever rvf-crypto ships the public API.
   This should not block K3's core deliverables but should be resolved before
   K4 to ensure the chain's post-quantum resilience is in place before
   container-based service calls generate high-volume chain entries.

## 7. Recommendations for K3

K3 ("WASM Signed Containers + ServiceApi") builds directly on the K2.1
foundation. The following items from the symposium are K3's primary input:

### 7.1 Must-Have for K3

1. **ServiceApi trait (C2)**: The internal API surface that protocol adapters
   bind to. This is the centerpiece of K3. The `ServiceEntry` metadata model
   from K2.1 is ready; K3 adds the runtime behavior.

2. **WASM sandbox runtime**: `SpawnBackend::Wasm` is defined and returns
   `BackendNotAvailable`. K3 implements Wasmtime integration so this variant
   works. This is the K3 headline feature.

3. **Dual-layer gate (C4)**: Add `GateBackend` to `A2ARouter::route()`. The
   `GovernanceGate` is already implemented and tested; C4 wires it into the
   routing path as a pre-delivery check.

4. **Chain-anchored contracts (C3)**: When services register their API schema
   via `ServiceApi::schema()`, append to ExoChain. The `ServiceEntry` and
   `ServiceRegistry` infrastructure is ready.

5. **N-dimensional EffectVector (C9)**: Refactor the 5D `EffectVector` to
   configurable dimensions. The `GovernanceGate` already extracts effect
   vectors from context JSON, so the refactor is localized.

### 7.2 Should-Have for K3

6. **Shell adapter for ServiceApi**: Map CLI args to `ServiceRequest`. This is
   the simplest protocol adapter and validates the `ServiceApi` trait shape.

7. **MCP adapter for ServiceApi**: Leverage the mcp-gate crate. This is the
   highest-value external protocol adapter.

8. **Post-quantum dual signing (C6)**: If rvf-crypto ships the DualKey API
   during K3 development, integrate it. If not, continue tracking.

### 7.3 Carry-Forward from Symposium

9. **Training data dimension (D18)**: K3 should add structured fields to chain
   events for ML consumption. Every new event type (service.*, tool.wasm.*)
   should include latency, fuel, and outcome fields.

10. **tiny-dancer routing hints (D17)**: Advisory routing layer in A2ARouter.
    K3/K4 timeframe per symposium. Can be K3 stretch goal or K4 definite.

11. **delta-consensus for off-chain state (D16)**: CRDT convergence for
    routing tables, health scores, capability caches. K3/K4 timeframe.

### 7.4 K3 Should NOT Include

- Container runtime (K4)
- Cross-node service discovery (K5)
- Blockchain anchoring implementation (K3/K4 -- trait only in K3)
- SONA integration (K5)
- TEE runtime (K6+)

### 7.5 K3 Entry Conditions (From K2.1)

K3 can begin immediately. All prerequisites are met:

- `SpawnBackend::Wasm` variant exists and returns clear error -> K3 implements it
- `MessageTarget::Service` routing works -> K3 adds ServiceApi behind it
- `ServiceEntry` metadata model is ready -> K3 adds runtime behavior
- `GovernanceGate` is tested end-to-end -> K3 wires it into A2ARouter (C4)
- `EffectVector` is 5D -> K3 refactors to N-dimensional (C9)
- Documentation updated with K2.1 changes and K3 expectations

crystallizations before K3, so K3 starts on a solid foundation. The only
genuine gap is C6 (post-quantum dual signing), which was blocked by an
upstream dependency and handled correctly as a documented deferral. All

follow-up items identified here are all properly phased (K3 or later)
and none represent missed K2.1 obligations except C6.
